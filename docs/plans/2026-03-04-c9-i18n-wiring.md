# C9 — i18n Wiring Plan

**Date:** 2026-03-04
**Audit ref:** C9 in `docs/audits/2026-03-04-full-stack-audit.md`
**Priority:** Tier 2 (before beta)

---

## What the Audit Got Wrong

The audit said `src/i18n/locales/` is empty. That directory doesn't exist — messages live in `app/messages/`. All three locale files exist and have real translations:

- `app/messages/en.json`
- `app/messages/pt-BR.json`
- `app/messages/es.json`

The infrastructure is also fully wired:
- `src/i18n/routing.ts` — defines `["en", "pt-BR", "es"]`, default `en`, prefix `as-needed`
- `src/i18n/request.ts` — `getRequestConfig` loads `../../messages/${locale}.json` on every request
- `src/i18n/navigation.ts` — locale-aware `Link`, `redirect`, `usePathname`, `useRouter`

## The Real Problem

Zero components call `useTranslations()` or `getTranslations()`. Every user-facing string is hard-coded English. The locale switcher changes the URL but the page text never changes.

## Current Message Coverage

7 namespaces, ~51 keys total across all three locales:

| Namespace | Keys | Coverage |

**Not covered at all:** Profile, Settings, Certificates, Lessons, course detail, lesson navigation, code challenge, quiz renderer, dashboard sub-components, landing features/CTA sections.

## Usage Patterns

### Client components (`"use client"`)

```ts
import { useTranslations } from "next-intl";

export function CourseCard() {
  const t = useTranslations("Courses");
  return <span>{t("startCourse")}</span>;
}
```

### Server components / page.tsx (async)

```ts
import { getTranslations } from "next-intl/server";

export default async function CoursesPage() {
  const t = await getTranslations("Courses");
  return <h1>{t("allCourses")}</h1>;
}
```

### Passing locale to metadata

```ts
export async function generateMetadata({ params }: { params: { locale: string } }) {
  const t = await getTranslations({ locale: params.locale, namespace: "Courses" });
  return { title: t("allCourses") };
}
```

**Note:** Client components cannot accept `t` as a prop from server components (serialization boundary). Each client component calls `useTranslations()` itself.

## Execution Notes

- Do not translate `en.json` strings that come from Sanity (course titles, lesson content) — those are CMS-managed
- `difficulty-badge.tsx` maps `"beginner" | "intermediate" | "advanced"` — these values come from Sanity but display labels should use `Courses.beginner` etc.
- `Common.loading` and `Common.error` should be used in all loading/error states, replacing the current hard-coded strings in error boundaries
- Page-level `generateMetadata` functions need `getTranslations` too — currently all return hard-coded English titles
